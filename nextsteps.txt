CONTEXT
================================================================================

This is a React 19 + TypeScript 5.7 SPA built with Vite 6.3 and Tailwind CSS 4.1.
It visualises Formula 1 session data consumed from the public OpenF1 REST API.
Charting is handled by Recharts 2.15. Routing is hash-based (no React Router).
Deployment target: GitHub Pages at /f1-telemetry-dashboard/.

Key files:
  src/api/openf1.ts                       — OpenF1 API client with caching & abort
  src/hooks/useOpenF1.ts                  — Generic data-fetching hook (cache TTL 5 min)
  src/hooks/useDashboardFilters.ts        — URL-hash state (year/circuit/session/drivers)
  src/hooks/useDashboardSelectionData.ts  — Derives available options from API data
  src/hooks/useDashboardViewModel.ts      — Transforms raw API data into chart models
  src/components/F1TelemetryDashboard.tsx — Root orchestrator component (~610 lines)
  src/components/dashboard/              — One file per tab + shared primitives

Branching convention:
  git checkout develop
  git checkout -b bugfix/<slug>    # for bug fixes
  git checkout -b feature/<slug>   # for features / perf / visual changes
  # After work: open PR to develop; after review merge develop -> main

## 1.1 Hash scroll timer not cleaned up on unmount
File: src/components/F1TelemetryDashboard.tsx  ~line 370

PROBLEM:
  The hash-scroll effect creates an interval with `window.setInterval` but does
  not capture the timer ID for cleanup. If the component unmounts while the
  interval is running the callback will still fire, causing a setState on an
  unmounted component (React 19 gives an error for this) and potentially
  scrolling to a stale element.

  Additionally the `attempts` counter is declared inside the callback closure
  via an outer let, which means if the hash changes and the effect re-runs, the
  previous interval is not cancelled and two intervals race.

FIX:
  1. Capture the timer ID returned by setInterval.
  2. Return a cleanup function from useEffect that calls clearInterval(timerId).
  3. Check signal.aborted before adding a listener:

  ```typescript
  useEffect(() => {
    if (!window.location.hash) return;
    const target = window.location.hash.slice(1);
    let attempts = 0;
    const timerId = window.setInterval(() => {
      const el = document.getElementById(target);
      if (el) {
        el.scrollIntoView({ behavior: 'smooth', block: 'start' });
        window.clearInterval(timerId);
        return;
      }
      attempts += 1;
      if (attempts >= 12) window.clearInterval(timerId);
    }, 120);
    return () => window.clearInterval(timerId);
  }, []);
  ```

## 1.2 Abort signal race condition in openf1.ts
File: src/api/openf1.ts  ~line 195-217

PROBLEM:
  `onAbort()` is registered with `signal.addEventListener('abort', onAbort)` but
  the signal may already be in the aborted state before the listener is attached
  (e.g. if abort() was called synchronously before the fetch resolves). This
  means the cleanup function is never called and the in-flight map leaks.

FIX:
  After registering the listener, immediately check `signal.aborted` and invoke
  cleanup if true:

  ```typescript
  signal.addEventListener('abort', onAbort, { once: true });
  if (signal.aborted) onAbort();
  ```

## 1.3 Math.min on empty array causes Infinity in TelemetryTab
File: src/components/dashboard/TelemetryTab.tsx  ~line 49-52

PROBLEM:
  ```typescript
  const bestS1 = Math.min(...sectorRows.map((row) => row.s1 ?? Number.POSITIVE_INFINITY));
  ```
  If `sectorRows` is empty, `Math.min()` with no arguments returns `Infinity`.
  This value propagates into the chart's Y-axis domain, producing undefined
  visual behaviour (blank charts or Recharts throwing warnings).

FIX:
  Guard with a length check and provide a sensible fallback:

  ```typescript
  const bestS1 = sectorRows.length
    ? Math.min(...sectorRows.map((r) => r.s1 ?? Number.POSITIVE_INFINITY))
    : null;
  ```
  Then gate chart rendering on `bestS1 !== null`.

## 1.4 Preferred lap selection falls through to invalid index
File: src/hooks/useDashboardSelectionData.ts  ~line 123-131

PROBLEM:

  fallback (`lapOptions[1]`) without checking whether `lapOptions` has at least
  two entries. If only one lap exists, this returns `undefined`, which is then
  used as a lap number and causes downstream API calls with `lap_number=undefined`.

FIX:
  ```typescript
  const preferred = lapOptions.includes(lapNum)
    ? lapNum
    : lapOptions[1] ?? lapOptions[0] ?? null;
  ```
  And propagate `null` cleanly rather than `undefined`.

## 2.1 Hardcoded 4-driver hooks prevent dynamic selection
File: src/components/F1TelemetryDashboard.tsx  ~line 186-245

PROBLEM:
  The component unconditionally calls `useLaps` and `useTelemetry` four times,
  one per driver slot, regardless of how many drivers are actually selected.
  React hooks cannot be called inside loops (Rules of Hooks), but they CAN be
  refactored by creating a fixed-length array of hooks:

  ```typescript
  // Current — fetches for driver[2] and driver[3] even if only 1 driver selected
  const laps0 = useLaps(sessionKey, driverNums[0]);
  const laps1 = useLaps(sessionKey, driverNums[1]);
  const laps2 = useLaps(sessionKey, driverNums[2]);
  const laps3 = useLaps(sessionKey, driverNums[3]);
  ```

FIX:
  Keep the same number of hook calls (required by Rules of Hooks) but pass
  `null` as the driver number when there is no driver in that slot, and make
  `useLaps`/`useTelemetry` skip the fetch when the driver is null:

  ```typescript
  // In useOpenF1.ts: skip fetch when driverNumber === null
  const MAX_DRIVERS = 4;
  const slots = Array.from({ length: MAX_DRIVERS }, (_, i) => driverNums[i] ?? null);
  const laps0 = useLaps(sessionKey, slots[0]);
  const laps1 = useLaps(sessionKey, slots[1]);
  const laps2 = useLaps(sessionKey, slots[2]);
  const laps3 = useLaps(sessionKey, slots[3]);
  ```
  This avoids unnecessary network requests while respecting Rules of Hooks.

## 2.3 O(n²) position bucketing in PositionsTab
File: src/components/dashboard/PositionsTab.tsx  ~line 26-69

PROBLEM:
  For each time bucket, the code searches through all position samples for each
  driver. Complexity is O(buckets × drivers × samples per driver). With a full
  race (~100 minutes × 20 drivers × many position updates) this is expensive.

FIX:
  Pre-index position samples by driver_number into a Map, then walk each
  driver's samples once in sorted order to fill buckets with a pointer
  per driver:

  ```typescript
  // O(n log n) for sort + O(n) for bucketing with pointer
  const byDriver = new Map<number, PositionSample[]>();
  allSamples.forEach((s) => {
    if (!byDriver.has(s.driver_number)) byDriver.set(s.driver_number, []);
    byDriver.get(s.driver_number)!.push(s);
  });
  // Sort once per driver, then advance pointers through buckets
  ```

## 5.1 Add Error Boundaries around each tab
File: src/components/F1TelemetryDashboard.tsx + new src/components/ErrorBoundary.tsx

FIX:
  Create `src/components/ErrorBoundary.tsx`:
  ```typescript
  class ErrorBoundary extends React.Component<
    { children: ReactNode; label: string },
    { error: Error | null }
  > {
    state = { error: null };
    static getDerivedStateFromError(error: Error) { return { error }; }
    render() {
      if (this.state.error) {
        return (
          <div className="flex flex-col items-center justify-center p-8 gap-4">
            <p className="text-red-400">Failed to render {this.props.label}</p>
            <button onClick={() => this.setState({ error: null })}>Retry</button>
          </div>
        );
      }
      return this.props.children;
    }
  }
  ```
  Wrap each `<*Tab … />` with `<ErrorBoundary label="Telemetry">`.

  ```typescript
    setError(null);
  } catch (err) {
    if (err instanceof Error && err.name === 'AbortError') return;
    setError(err instanceof Error ? err : new Error(String(err)));
    setData(null);
  } finally {
    setLoading(false);
  }
  ```

## 7.1 Split F1TelemetryDashboard into Container + Presenter

## 7.2 Unified state layer
Files: useDashboardFilters.ts + useDashboardSelectionData.ts + useDashboardViewModel.ts

  The three hooks depend on each other in a chain (filters → selectionData → viewModel)
  but are split into three files making the data flow hard to follow.

  Consolidate into `useDashboard()` which internally composes the three:
  ```typescript
  export function useDashboard() {
    const filters = useDashboardFilters();
    const selection = useDashboardSelectionData(filters);
    const viewModel = useDashboardViewModel(filters, selection);
    return { filters, selection, viewModel };
  }
  ```
  Keep each sub-hook private (not exported) to enforce the single entry point.

## 7.3 Request deduplication in openf1.ts
File: src/api/openf1.ts

  The `inflight` map deduplicates in-flight requests keyed by URL. However if
  two components unmount and abort concurrently, the second abort could cancel
  the cache entry before the first consumer receives it.

  FIX:
  Use a reference-counted abort controller: only abort the underlying fetch when
  the last subscriber cancels. This prevents one tab's unmount from cancelling
  data another tab still needs.
